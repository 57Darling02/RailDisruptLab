# 扰动图生成框架

## 0. 读文档顺序和当前结论

如果从零接手项目，建议按顺序读：

```text
docs/框架优化.md       -> 理解研究目标、边界和核心设计
docs/框架实现Plan.md   -> 理解当前代码状态、接口和待验证项
docs/exp.md            -> 按命令跑完整实验和验收流程
```

当前结论：

- 工程目标是生成“铁路扰动事实”，不是生成 MILP 矩阵。
- 主程序 RailGraph2Gurobi 是唯一铁路语义源。
- VAE 只读数学规则和数值图，只生成数学 task output。
- VAE 当前代码已扁平化在 `VAE/src`，旧 ACM-MILP 流程已清理。
- 当前 VAE 已使用 `edge_index + edge_attr` 做 typed message passing，不再只是把节点特征做均值池化。
- 本地已验证非 torch 链路：导出 math dataset、target-copy 生成、解码、回灌、build。
- 真实 VAE forward/backward、短训练、model generate 尚未在本机验证，因为当前本机没有 torch 环境。

## 1. 当前目标

本框架不让 VAE 生成 MILP 约束矩阵，也不让 VAE 直接理解铁路业务语义。RailGraph2Gurobi 先把铁路语义编译为数学规则和数值图，VAE 在这个数学对象上学习潜在规律并生成新的数学结果，最后由 RailGraph2Gurobi 恢复为铁路扰动。

主链路：

```text
RailGraph2Gurobi:
  BaseContext + scenario config
    -> vae_math_learning_graph
    -> vae_math_dataset_profile

VAE:
  vae_math_learning_graph
    -> train
    -> vae_math_generated_graph

RailGraph2Gurobi:
  vae_math_generated_graph
    -> disturbance_graph
    -> scenario config
    -> build / solve / export-timetable / analyze
```

## 2. 语义与数学边界

RailGraph2Gurobi 负责：

- 识别合法锚点。
- 判断扰动类型和作用对象。
- 生成 pool、edge、task、parameter 的数学规则。
- 将 VAE 输出的 task id、pool index、parameter vector 解码回铁路扰动。
- 做时间窗、速度阈值、中断解释和 MILP 验证。

VAE 负责：

- 读取数值矩阵和数值规则。
- 学习 count、anchor index、params 的联合分布。
- 生成 `vae_math_generated_graph`。

VAE 不读取：

- 车次、站名、区间名。
- event/section/delay/speed 等业务字符串。
- anchor id。
- debug 字段。
- feature name 或 param name。
- scenario YAML 或 disturbance graph。

## 3. 数学对象

数学主文件的核心是：

```text
C:
  rules.pools
  rules.edge_types
  rules.tasks
  graph.pool_x
  graph.edges

G_D:
  supervision.targets

R:
  supervision.target_relations
```

其中：

- `C` 是题目，即上下文规则和数值图。
- `G_D` 是标准答案，即真实扰动的 count、anchor index、params。
- `R` 是训练辅助关系，只作为 encoder/auxiliary 信息，不由 decoder 生成。

### 3.1 节点和边设计

当前数学图里有两个候选池，也可以理解为两类节点：

- pool 0: event anchor，表示“某车次在某站的到达/出发事件”。
  它的数值特征包括计划时间、车次序号、站序号、方向、站点顺序、到发类型。
- pool 1: section anchor，表示“一个运行区间”。
  它的数值特征包括区间顺序、里程、最小运行时分、方向。

边有三类：

- event-event edge：连接相关事件，例如同车、同站、同方向、时间接近、站序接近。
- section-section edge：连接相关区间，例如共享端点、相邻区间、同方向、里程/区间序接近。
- event-section edge：连接事件和区间，例如事件站是否是区间端点、车次路径是否经过该区间。

这些边不是铁路语义字符串，而是数值关系。VAE 只看到 pool id、index、edge type 和 edge feature。

### 3.2 GNN 消息传递实现细节

这里不是直接把原始同维度节点相加。event 节点原始特征是 6 维，section 节点原始特征是 4 维；模型先用各自的 encoder 把它们投影到同一个 `hidden_dim`。之后所有消息传递都发生在这个统一隐空间里。

边也不是只表示连通性。每条边有自己的 `edge_attr`，例如：

```text
event A -- 同车/同站/时间接近 --> event B
event A -- 车次路径经过区间 --> section S
section S -- 相邻区间 --> section T
```

一次消息传递可以理解为：

```text
source node hidden + edge hidden
  -> type-specific message layer
  -> message
  -> 按 edge_index 加到 target node 的收件箱
```

节点收到多个消息后，不是简单求和，而是先按入边数量求平均，再和自己的旧表示一起进入 pool-specific update layer：

```text
agg = mean(incoming messages)
new_node = LayerNorm(old_node + update_layer([old_node, agg]))
```

当前三类边各自有独立参数：

- event-event: 在事件节点之间传。
- section-section: 在区间节点之间传。
- event-section: 在事件节点和区间节点之间跨池传。

当前导出的上下文边是无向语义边，因此模型会同时做 forward 和 reverse 两个方向的消息。重复 2 轮后，一个 event anchor 不只知道自己的计划时间，也能知道附近有哪些同车事件、同站事件、相关区间；一个 section anchor 也能知道相邻区间和经过它的事件。这样 decoder 在选择扰动落点时用到的是局部结构信息，而不是孤立节点信息。

## 4. 分池预测器

每个 task 都有一个纯数学规则：

```text
task_id -> target_pool_id
```

decoder 对某个 task 预测 anchor 时，只在该 task 的 `target_pool_id` 对应 pool 上打分。这避免了让模型学习“哪类扰动可以作用在哪类锚点”的业务语法。

当前 v1：

```text
task 0 -> pool 0 -> param_dim 1
task 1 -> pool 1 -> param_dim 3
```

这些编号在 VAE 内部没有业务含义。业务解释只存在于 RailGraph2Gurobi 和 `dataset_profile.json` 中。

## 5. max_slots

`max_slots` 是每个 task 的最大生成槽位数，是固定张量上限，不是铁路语义。

例如：

```text
max_slots = 8
count = 2
```

表示该样本真实只有前 2 个 slot 有监督；slot 2 到 slot 7 是工程 padding，不参与 anchor/param loss。count 本身仍参与监督。

## 6. Dataset Profile

`dataset_profile.json` 与一个 `base_context_path` 绑定，只生成一份。它保留：

- pool 行号到 anchor id 的映射。
- feature 列名。
- task 和 parameter 的可读说明。
- source config 列表。
- exporter 参数。
- decode contract。
- debug 信息。

profile 的作用是排查和解释，不是 VAE 输入。VAE reader 必须只接受 `vae_math_learning_graph`。

## 7. Louvain 定位

当前不把 Louvain 社区发现和局部替换放入主流程。结构相似性和求解难度保持作为评估层保留：VAE 侧比较 `vae_math_learning_graph` 与 `vae_math_generated_graph` 的数值分布，求解难度在 RailGraph2Gurobi decode/import/build/solve 后用 `bench_solve.py --summary-csv` 对比，不改变数学主接口的基本边界。
