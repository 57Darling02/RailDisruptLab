# 扰动图生成框架

这篇文档只讲框架思路。它不记录脚本参数、不记录工程入口，也不做实验清单；这些内容分别放在 `docs/框架实现Plan.md` 和 `docs/exp.md`。

这里要说明的是：为什么不让 VAE 直接生成 MILP，为什么要把铁路扰动先改写成数学图，以及这张数学图在数据里具体是什么样子。

建议按这个顺序读：

## 1. 核心目标

这个项目现在的目标不是让 VAE 学会铁路业务规则，也不是让 VAE 直接生成 MILP 约束矩阵。更稳的做法是：

```text
RailGraph2Gurobi 先把铁路问题整理成数学图
VAE 在数学图上学习扰动的落点和参数
RailGraph2Gurobi 再把 VAE 输出还原成铁路扰动
```

这样分工后，铁路语义只有一个来源：RailGraph2Gurobi。VAE 不需要知道车次名、站名、区间名，也不需要知道 YAML 怎么写、MILP 怎么建。它只看数值矩阵和规则编号。

主链路可以写成：

```text
RailGraph2Gurobi:
  BaseContext + scenario config
    -> 一个共享上下文 C
    -> 多个样本答案 G_D / R
    -> 一个 dataset profile

VAE:
  训练时读取 C + G_D / R
  生成时只读取 C
    -> 生成数学 task output

RailGraph2Gurobi:
  读取 task output
    -> disturbance_graph
    -> scenario config
    -> build / solve / export-timetable / analyze
```

## 2. 语义边界

边界可以简单理解为：

```text
RailGraph2Gurobi 负责“这是什么铁路对象”
VAE 负责“在这些候选对象上，扰动更可能怎么分布”
```

RailGraph2Gurobi 负责：

- 从 BaseContext 中确定合法 event anchor 和 section anchor。
- 把候选对象转成数值特征。
- 把候选对象之间的关系转成数值边。
- 定义 task 可以作用在哪个候选池上。
- 把 VAE 输出的 task、pool index、参数向量解码回铁路扰动。
- 做时间窗、速度阈值、中断解释和 MILP 验证。

VAE 负责：

- 读取数值节点、数值边和 task 规则。
- 学习扰动数量、锚点位置和参数。
- 生成新的数学输出。

VAE 不读取：

- 车次、站名、区间名。
- anchor id。
- feature name 和 param name。
- scenario YAML。
- disturbance graph。
- MILP 变量、约束或矩阵。

这个边界的好处是代码比较干净：铁路规则不散落到 VAE 里，VAE 输出也不会绕过主程序直接改业务对象。

## 3. 三类数据

当前接口里有三类核心数据：

```text
C   = vae_math_context_graph
G_D = vae_math_learning_sample 里的 supervision.targets
R   = vae_math_learning_sample 里的 supervision.target_relations
```

可以把它们理解成：

- `C` 是题目：有哪些候选锚点、它们有什么数值特征、它们之间有什么关系、每个 task 应该在哪个候选池里选。
- `G_D` 是标准答案：真实样本中有几个扰动、落在哪些候选锚点上、参数是多少。
- `R` 是训练辅助关系：真实扰动之间是否空间接近、时间重叠等。它帮助训练，但生成结果里不直接输出它。

训练时是一份共享题目配多份答案：

```text
context.json
  C

graph_samples/case0001.json
  G_D / R

graph_samples/case0002.json
  G_D / R
```

生成时只有题目：

```text
context.json
  C

VAE output
  task_outputs
```

这样做是因为同一个基本运行图下，候选锚点和候选关系是不变的。不同样本变化的是扰动事实，也就是答案部分。

## 4. 上下文 C 长什么样

`C` 是共享上下文图。它在文件里大致是：

```json
{
  "graph_type": "vae_math_context_graph",
  "decode_handle": {},
  "rules": {
    "pools": [],
    "edge_types": [],
    "tasks": []
  },
  "graph": {
    "pool_x": {},
    "edges": []
  }
}
```

这里没有铁路语义字符串。VAE 看到的是编号和数值。

### 4.1 候选池

当前有两个候选池：

```text
pool 0 = event anchor
pool 1 = section anchor
```

event anchor 表示“某车次在某站的到达/出发事件”。它的原始特征是 6 维：

```text
[
  planned_time_norm,
  train_index_norm,
  station_index_norm,
  direction_id,
  station_order_norm,
  event_type_id
]
```

section anchor 表示“一个运行区间”。它的原始特征是 4 维：

```text
[
  section_order_norm,
  mileage_norm,
  min_runtime_norm,
  direction_id
]
```

写到 `context.json` 里，就是按 pool 分开的矩阵：

```json
"pool_x": {
  "0": [
    [0.31, 0.02, 0.10, 0.0, 0.12, 1.0],
    [0.34, 0.02, 0.15, 0.0, 0.18, 0.0]
  ],
  "1": [
    [0.10, 0.08, 0.05, 0.0],
    [0.20, 0.14, 0.06, 0.0]
  ]
}
```

矩阵的行号就是 VAE 用的候选 index。真正的 anchor id 只保存在 `dataset_profile.json` 里，方便 RailGraph2Gurobi 解码和人排查。

### 4.2 关系边

候选对象之间还有关系边。当前有三类：

```text
edge type 0 = event-event
edge type 1 = section-section
edge type 2 = event-section
```

event-event 边描述事件之间的关系，比如同车、同站、时间接近。当前特征是 3 维：

```text
[same_train, same_station, planned_time_diff_norm]
```

`planned_time_affinity` 仍可作为内部选边排序分数，但不再进入 edge vector。

section-section 边描述区间之间的拓扑邻近。当前特征是 1 维：

```text
[mileage_diff_norm]
```

`shared_endpoint`、`adjacent_section` 和 `section_order_diff_norm` 主要由边存在和节点两端特征表达，不再重复写入向量。

event-section 边描述事件和区间之间的关系，比如该车次路径是否经过这个区间、事件站到区间端点的距离。当前特征是 2 维：

```text
[train_route_contains_section, station_to_section_order_distance_norm]
```

`event_station_is_section_endpoint` 等价于距离为 0，`same_direction` 由两端节点的 `direction_id` 交给 message layer 自己比较。

写到文件里，一条边类似这样：

```json
{
  "edge_type_id": 2,
  "source_pool_id": 0,
  "source_index": 15,
  "target_pool_id": 1,
  "target_index": 3,
  "directed": false,
  "x": [1.0, 0.0]
}
```

这里的意思很直接：pool 0 的第 15 个 event anchor 和 pool 1 的第 3 个 section anchor 有一条 event-section 关系，关系本身用 2 个数描述。`directed: false` 表示这条关系可以双向传消息。

## 5. 答案 G_D / R 长什么样

每个训练样本只保存答案，不重复保存上下文：

```json
{
  "graph_type": "vae_math_learning_sample",
  "context_ref": "context.json",
  "supervision": {
    "targets": {},
    "target_relations": []
  }
}
```

`targets` 按 task 保存。当前有两个 task：

```text
task 0 = 在 event pool 中生成 delay
task 1 = 在 section pool 中生成 speed_limit / interruption
```

一个简化的答案长这样：

```json
"targets": {
  "0": {
    "count": 1,
    "anchor_index": [15],
    "params": [[0.0069]]
  },
  "1": {
    "count": 2,
    "anchor_index": [3, 8],
    "params": [
      [0.35, 0.04, 0.22857143],
      [0.42, 0.02, 0.0]
    ]
  }
}
```

含义是：

- task 0 生成 1 个 delay，落在 event pool 的第 15 行，参数是归一化后的延误时长。
- task 1 生成 2 个区间类扰动，落在 section pool 的第 3 行和第 8 行，参数是开始时间、持续时间和归一化限速值。
- 当前限速按 `speed_limit_norm = speed_limit / 350` 写入样本；例如 80 km/h 写成 `0.22857143`。
- 归一化限速解码回真实限速后，如果低于中断阈值，就解释为中断。

`target_relations` 描述答案之间的数学关系，例如两个扰动是否时间重叠、空间接近。它不是生成目标，只是训练时给 encoder 多一点结构信息。

## 6. VAE 怎么看这张图

VAE 不是把所有节点简单求平均。它先让节点沿着边互相看一眼，再汇总整张图。

过程可以用平实的话说：

```text
先把 event 节点、section 节点、边特征都变成同样长度
再按 edge_index 找到谁连着谁
把 source 节点、target 节点和边的信息一起计算消息
每个节点收到邻居消息后更新自己的表示
最后把所有节点和边汇总成一个 context 向量
```

数据形状上大致是：

```text
event 6维        -> encoder -> hidden_dim
section 4维      -> encoder -> hidden_dim
edge feature     -> encoder -> hidden_dim

[source_hidden, target_hidden, edge_hidden] -> message -> target 聚合
[old_node_hidden, 聚合消息]  -> update  -> new_node_hidden
```

默认 `hidden_dim = 64`，所以当前上下文汇总后是：

```text
2 个 pool 向量 + 3 类 edge 向量
= 5 * 64
= 320 维 context 向量
```

这 320 维不是业务文本，而是模型看到的“当前运行图摘要”。

训练时，模型同时看上下文和标准答案：

```text
C + G_D / R -> posterior -> z -> decoder
```

生成时，没有标准答案，只看上下文：

```text
C -> prior -> z -> decoder
```

所以 `C` 是固定背景，`G_D / R` 是训练阶段才有的答案信息。

## 7. Decoder 怎么选锚点

每个 task 都绑定一个目标 pool：

```text
task 0 -> pool 0 -> delay 参数 1 维
task 1 -> pool 1 -> section 参数 3 维
```

decoder 不会在整张图里随便选。它按 task 做三件事：

```text
预测 count
为每个 slot 生成一个 query
只在该 task 的目标 pool 中给候选 anchor 打分
```

例如 task 0 只会在 event pool 里选，task 1 只会在 section pool 里选。这样业务语法仍然由 RailGraph2Gurobi 编译成规则，VAE 只负责在规则给定的候选池里做选择。

参数也是同样的思路：VAE 生成数值向量，RailGraph2Gurobi 负责把这些数值解释成延误秒数、开始时间、持续时间、限速或中断。

## 8. 生成结果是什么

VAE 输出的是数学结果：

```json
{
  "graph_type": "vae_math_generated_graph",
  "decode_handle": {},
  "task_outputs": {
    "0": {
      "count": 1,
      "anchor_index": [15],
      "params": [[0.0069]]
    },
    "1": {
      "count": 1,
      "anchor_index": [3],
      "params": [[0.35, 0.04, 0.22857143]]
    }
  }
}
```

它不是最终业务配置。RailGraph2Gurobi 会继续做：

```text
vae_math_generated_graph
  -> disturbance_graph
  -> scenario config
  -> build / solve / export-timetable / analyze
```

这一步很重要，因为参数边界修正、24 小时时间窗、限速阈值、中断解释和 MILP 验证都应该由主程序统一处理。

## 9. Dataset Profile 的位置

`dataset_profile.json` 是解释文件，不是 VAE 输入。

它保留人能看懂的信息：

- pool 行号到 anchor id 的映射。
- feature 名称。
- task 和参数说明。
- source config 列表。
- decode contract。
- debug 信息。

VAE 训练只读 `context.json` 和 `graph_samples/*.json`。profile 的作用是排查问题和给 RailGraph2Gurobi 解码流程提供解释，不参与模型学习。

## 10. 当前不放进主流程的内容

Louvain 社区发现和局部替换暂时不放进主链路。它们更适合留在评估或后处理探索里。

当前主流程保持简单：

```text
数学图生成 -> VAE 训练/生成 -> 解码回铁路扰动 -> MILP 验证
```

结构相似性和求解难度也放在评估层，而且要分层看：

- VAE task output 层：比较 count、anchor 分布、参数分布。
- 扰动图层：比较扰动节点数、kind 组合、role edge 数和时间/空间关系。
- MILP 实例层：比较 build 后变量数、约束数、非零项数。
- solver 层：在相同 time limit 下比较 status、duration、objective 和 MIP gap。

同一个 BaseContext 下，VAE context graph 的候选节点数是固定的，因此不能只拿候选节点数判断模型是否学到位。更应该看生成扰动图规模、MILP 实例规模和求解行为是否接近 reference。

这样主接口不被额外方法绑死，后续要换模型或加评估指标也比较容易。
