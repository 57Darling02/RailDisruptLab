# Typed Graph CVAE 模型方法说明

本文档面向论文写作，描述当前 RailGraph2Gurobi -> VAE -> RailGraph2Gurobi 链路中的模型抽象。工程入口、脚本参数和验收步骤不放在这里，见 `docs/框架实现Plan.md` 和 `docs/exp.md`。

## 1. 条件生成问题

本项目不让神经网络直接生成 MILP 变量、约束或业务 YAML，而是把扰动生成建模为一个带类型约束的条件生成问题。

给定一个固定运行图上下文：

```text
C = (V, E, X, A, T)
```

模型学习生成扰动答案：

```text
Y = {Y_t | t in T}
```

其中：

- `V` 是候选锚点集合，但按类型拆成多个 pool。
- `E` 是候选锚点之间的关系边，也带 edge type。
- `X` 是节点数值特征。
- `A` 是边数值特征。
- `T` 是生成 task 规则。
- `Y_t` 是 task `t` 的扰动数量、锚点选择和参数。

训练目标不是重建整个上下文图。上下文 `C` 在同一个 BaseContext 下固定，模型需要学习的是：在这个上下文条件下，扰动通常落在哪里、数量是多少、参数如何分布。

因此训练样本可以写成：

```text
(C, Y, R)
```

其中 `R` 是扰动之间的辅助关系特征，只用于训练阶段的 posterior encoder；生成阶段只使用 `C`。

## 2. Typed Context Graph

上下文图不是一个同质图，而是 typed graph。当前有两个节点 pool：

```text
pool 0: event anchor
pool 1: section anchor
```

event anchor 表示某车次在某站的一次到达或出发事件。它的原始特征为：

```text
x_event = [
  planned_time_norm,
  train_index_norm,
  station_index_norm,
  direction_id,
  station_order_norm,
  event_type_id
]
```

section anchor 表示一个运行区间。它的原始特征为：

```text
x_section = [
  section_order_norm,
  mileage_norm,
  min_runtime_norm,
  direction_id
]
```

节点按 pool 保存为矩阵：

```text
X_p in R^(n_p x d_p)
```

其中 `p` 是 pool id，`n_p` 是该 pool 的候选数量，`d_p` 是该 pool 的原始特征维度。矩阵行号就是模型用于选择 anchor 的 index。真实铁路对象 id 不进入模型，只保留在 profile 和 decode handle 中。

## 3. Typed Edges

边也按类型定义。当前有三类：

```text
edge type 0: event-event
edge type 1: section-section
edge type 2: event-section
```

一条边可以表示为：

```text
e = (p_s, i, p_t, j, r, a_e, directed)
```

含义是：source pool `p_s` 的第 `i` 个节点，与 target pool `p_t` 的第 `j` 个节点之间存在 edge type `r` 的关系，边特征为 `a_e`。

当前 edge feature 经过精简后为：

```text
event-event:
  [same_train, same_station, planned_time_diff_norm]

section-section:
  [mileage_diff_norm]

event-section:
  [train_route_contains_section, station_to_section_order_distance_norm]
```

这里的设计原则是：边存在和边类型表达拓扑关系，边向量只保留少数不容易由两端节点直接替代的连续强度或关系标记。方向、站序、端点等可由 source/target 节点特征与边存在共同表达的信息，不再重复堆入 edge vector。

## 4. GNN Context Encoder

不同 pool 的原始特征维度不同，所以模型先用类型专属 encoder 投影到统一隐空间：

```text
h_i^(p,0) = f_p(x_i^p)
g_e       = f_r(a_e)
```

其中：

- `h_i^(p,0)` 是 pool `p` 中第 `i` 个节点的初始隐表示。
- `g_e` 是边 `e` 的隐表示。
- `f_p` 是 pool-specific MLP。
- `f_r` 是 edge-type-specific MLP。

每一轮 message passing 按 edge type 分开计算。对于边：

```text
e = (p_s, i, p_t, j, r, a_e)
```

正向消息为：

```text
m_e^forward = psi_r^forward(
  [h_i^(p_s,l), h_j^(p_t,l), g_e]
)
```

如果该边是无向语义边，还额外计算反向消息：

```text
m_e^reverse = psi_r^reverse(
  [h_j^(p_t,l), h_i^(p_s,l), g_e]
)
```

注意当前 message MLP 显式读取 source node、target node 和 edge 三部分。这一点很关键：模型可以在 message 函数内部比较两端节点表示，因此不需要把大量 equality、direction、order diff 等派生关系重复塞进 edge vector。

每个节点收到的消息做平均聚合：

```text
bar_m_i^(p,l) = mean({m_e -> i})
```

节点更新为：

```text
h_i^(p,l+1) = LayerNorm(
  h_i^(p,l) + u_p([h_i^(p,l), bar_m_i^(p,l)])
)
```

其中 `u_p` 是 pool-specific update MLP。重复 `L = message_passing_steps` 轮后，得到最终节点表示 `h_i^(p,L)`。

## 5. Message Passing Steps 的意义

`message_passing_steps` 控制节点表示的感受野。

```text
L = 0: 不使用图邻接，只编码节点自身特征
L = 1: 节点能接收直接邻居信息
L = 2: 节点能间接吸收邻居的邻居信息
L >= 3: 感受野继续扩大，但可能带来过平滑和更高训练成本
```

在本项目中，`L=2` 已能覆盖常见的间接关系，例如：

```text
event -> section -> event
section -> event -> section
```

更大的 `L` 是否有效，需要通过消融实验判断。消融不能只看训练 loss，还要看生成扰动图、build 后 MILP 规模和 solve 行为是否更接近 reference。

## 6. Context Vector

GNN 更新节点后，模型将每个 pool 的节点表示做 mean pooling：

```text
c_p = mean_i h_i^(p,L)
```

同时，每类边的边表示也做 mean pooling：

```text
c_r = mean_{e in E_r} g_e
```

最终上下文向量为：

```text
c = concat({c_p}, {c_r})
```

当前有 2 个 pool 和 3 类 edge type，因此：

```text
dim(c) = hidden_dim * (2 + 3)
```

当 `hidden_dim = 64` 时，`dim(c) = 320`。

## 7. Target Encoder

训练阶段，模型还要编码真实答案 `Y` 和辅助关系 `R`，用于构造 posterior。

每个 task 的监督由三部分组成：

```text
Y_t = (k_t, I_t, Theta_t)
```

其中：

- `k_t` 是该 task 的扰动数量。
- `I_t` 是被选中的 anchor index 序列。
- `Theta_t` 是每个 slot 对应的参数向量。

因为神经网络需要固定形状张量，训练样本会用 `max_slots` 做 padding。超过真实 `k_t` 的 slot 只是占位，不参与 anchor 和 param loss。

target encoder 输入大致为：

```text
[
  k_t / max_slots,
  padded_anchor_index / pool_size,
  padded_params
]
```

每个 task 单独编码后拼接。辅助关系 `R` 也经过 relation encoder 后做 pooling，并并入 target 表示：

```text
y = concat({target_encoder_t(Y_t)}, relation_encoder(R))
```

`R` 只参与训练时的 posterior，不作为生成结果输出。

## 8. Conditional VAE

模型采用条件 VAE 结构。

prior 只看上下文：

```text
p_theta(z | C) = Normal(mu_p(c), diag(sigma_p(c)^2))
```

posterior 同时看上下文和真实答案：

```text
q_phi(z | C, Y, R) = Normal(mu_q([c, y]), diag(sigma_q([c, y])^2))
```

训练时从 posterior 采样：

```text
z = mu_q + sigma_q * epsilon
```

生成时从 prior 采样：

```text
z = mu_p + sigma_p * epsilon
```

训练损失由两部分组成：

```text
reconstruction loss + beta * KL(q_phi(z | C,Y,R) || p_theta(z | C))
```

当前 reconstruction loss 分为：

```text
count_loss:  扰动数量分类损失
anchor_loss: 锚点选择分类损失，只计算真实 count 内 slot
param_loss:  参数回归损失，只计算真实 count 内 slot
```

总损失为：

```text
loss =
  count_weight * count_loss
  + anchor_weight * anchor_loss
  + param_weight * param_loss
  + kl_weight * KL
```

## 9. 分池 Decoder

decoder 输入为：

```text
d = D([z, c])
```

每个 task 都绑定一个目标 pool：

```text
task 0 -> event pool
task 1 -> section pool
```

因此 decoder 不是在全图所有节点上自由选择，而是在 task 指定的候选池里选择 anchor。这相当于把类型约束显式编译进生成空间。

对每个 task，decoder 输出三类结果：

```text
count_logits_t
slot_queries_t
params_t
```

每个 slot 的 anchor 选择通过 query 与目标 pool 节点表示做点积：

```text
anchor_logits_t = Q_t * H_{pool(t)}^T
```

其中：

- `Q_t` 是 task `t` 的 slot query 矩阵。
- `H_{pool(t)}` 是目标 pool 的节点表示矩阵。

这种做法有两个好处：

- task 类型约束天然成立，例如 delay 只能落在 event anchor 上。
- anchor 打分使用的是 GNN 更新后的节点表示，因此锚点选择可以利用图邻域信息。

参数向量由 task-specific parameter head 输出。解码回铁路扰动时，RailGraph2Gurobi 再负责参数边界、时间窗、限速阈值和中断解释。

## 10. 生成与回灌

VAE 生成的不是业务配置，而是数学 task output：

```text
vae_math_generated_graph =
{
  task_id -> {
    count,
    anchor_index,
    params
  }
}
```

RailGraph2Gurobi 使用 `decode_handle` 和 `dataset_profile` 中的映射，把 pool index 还原成 event anchor 或 section anchor，再构造标准扰动图：

```text
vae_math_generated_graph
  -> disturbance_graph
  -> scenario config
  -> MILP build
  -> solve / export / analyze
```

这个设计避免神经网络直接绕过业务规则。模型只负责生成数值选择，合法性解释和最终验证仍由确定性程序完成。

## 11. 评估指标

判断模型是否“学到位”，不能只看训练 loss。当前更合理的评估分三层。

第一层是数学 task output 相似性：

- count 分布是否接近 reference。
- task 组合是否接近 reference。
- anchor index 分布是否接近 reference。
- 参数分布是否接近 reference。

第二层是解码后的扰动图结构相似性：

- 扰动节点数是否接近 reference。
- delay / speed_limit / interruption 的比例是否接近 reference。
- role edge 数量和扰动落点类型是否合理。
- 扰动之间的时间重叠、空间邻近和 route 相关关系是否接近 reference。

第三层是 RailGraph2Gurobi build/solve 后的实例难度相似性：

- build 后变量数、约束数、非零项数是否接近 reference。
- 在相近规模实例上，solve status、solve duration、objective、MIP gap 是否接近 reference。
- 在固定 time limit 下，timeout 比例是否接近 reference。

这里要注意：VAE context graph 的候选节点数在同一个 BaseContext 下是固定的，不能直接用它判断生成质量。更有意义的是比较生成扰动图的节点/边规模，以及 build 后 MILP 实例的规模和求解行为。

## 12. Message Passing 消融应回答的问题

`message_passing_steps` 消融的核心问题是：多远的图邻域对扰动生成有帮助。

对比 `L=2/3/4` 时，应保持以下条件一致：

- 相同 reference dataset。
- 相同 hidden dimension 和 latent dimension。
- 相同训练 epoch、batch size、learning rate 和 seed。
- 相同生成样本数。
- 相同 decode/build/solve 流程。
- 相同 solver time limit。

然后比较：

- 训练收敛速度和 best loss。
- 生成 task output 与 reference 的分布距离。
- 解码后的扰动图结构是否接近 reference。
- build 后 MILP 规模是否接近 reference。
- solve 时间和 timeout 分布是否接近 reference。

如果更大的 `L` 只降低训练 loss，但生成结构和求解难度没有更接近 reference，则不应认为它更好。相反，如果 `L=2` 在结构和求解层面已经接近 reference，而 `L=3/4` 训练更慢或生成更偏，则应选择更简单的 `L=2`。
