# 扰动图理论

这篇文档只讲一件事：为什么要把列车扰动建成图，以及这张图如何接到 GNN+VAE 上。

GNN 和 VAE 本身都是成熟方法，不是本项目的创新点。本项目真正要解决的是：列车扰动不是规整图片、不是普通表格，也不是可以直接生成的 MILP 矩阵。它需要先被整理成一种适合学习、适合生成、也能被铁路规则重新解释的数据形式。

## 1. 问题判断

列车扰动调整实例有一个明显特点：

```text
规则性强，但规律性不强。
```

意思是：

- 扰动类型很多，组合方式也多。晚点、限速、中断、多个扰动叠加，发生位置和发生时间都可能不同。
- 不同扰动遵守的业务规则完全不同。晚点落在事件上，限速和中断落在区间上，参数含义也不同。
- 但一旦扰动类型确定，根据规则就能很清楚地推出它应该落在哪里、需要什么参数、会形成什么约束。

所以这里不能指望模型直接“学会铁路规则”。更合理的分工是：

```text
RailDisruptLab 负责规则和解释
VAE 负责学习扰动在规则空间里的分布
```

也就是说，业务语法先由 RailDisruptLab 编译成图和 task 规则。VAE 只在这个规则给定的空间里学习：更可能有几个扰动、落在哪些候选对象上、参数大概是什么。

## 2. 为什么不是直接生成 MILP

MILP 是最终求解形式，不是好的生成对象。

同一个扰动在 MILP 里会展开成大量变量、约束和系数。矩阵规模受建模方式影响很大，里面混有很多求解器需要的结构，但这些结构不等于扰动本身。如果让 VAE 直接生成 MILP，模型要同时学铁路语义、建模规则、矩阵布局和数值系数，边界会很乱。

本项目采用更稳的链路：

```text
计划时刻表 + 里程表 + 场景
  -> 扰动数学图
  -> VAE 学习/生成
  -> 解码回铁路扰动
  -> MILP build / solve / analyze
```

这样 MILP 仍然由确定规则生成，VAE 不绕过主程序直接改约束。

## 3. 为什么需要图

VAE 擅长学习分布和生成样本。问题是，传统图像天然规整：每张图都是固定网格，每个像素位置有灰度或 RGB。列车扰动没有这种天然网格。

扰动数据里有几类对象：

- 某车次在某站的到发事件。
- 某两个站之间的运行区间。
- 事件之间的时间、同车、同站关系。
- 区间之间的里程、拓扑关系。
- 事件和区间之间的线路经过关系。
- 真实扰动之间的时间重叠、空间接近、连锁关系。

这些对象数量不固定，关系也不规则。图结构正好适合表达这种数据：节点表示候选对象，边表示候选对象之间的关系，节点和边的属性全部转成数值向量。GNN 再把这些不规则关系聚合成模型能用的规整表示。

## 4. 三张图

当前设计里可以把数据理解成三张图。

### 4.1 上下文图 C

上下文图是全局背景，来自原计划时刻表和里程表。它不描述某一次具体扰动，而是描述“有哪些合法候选锚点，以及这些锚点之间是什么关系”。

```text
C = vae_math_context_graph
```

它包含两类候选池：

```text
pool 0 = event anchor
pool 1 = section anchor
```

event anchor 表示某车次在某站的到达或出发事件。当前特征是：

```text
[
  planned_time_norm,
  train_index_norm,
  station_index_norm,
  direction_id,
  station_order_norm,
  event_type_id
]
```

section anchor 表示一个运行区间。当前特征是：

```text
[
  section_order_norm,
  mileage_norm,
  min_runtime_norm,
  direction_id
]
```

上下文图还有三类边：

```text
edge type 0 = event-event
edge type 1 = section-section
edge type 2 = event-section
```

它们分别表达：

- event-event：同车、同站、计划时间差。
- section-section：区间里程差。
- event-section：该车次是否经过该区间、事件站点到区间端点的顺序距离。

这些边不是扰动本身，而是让 GNN 知道候选对象之间怎样相互影响。

### 4.2 目标扰动图 G_D

目标扰动图是模型真正要生成的对象。为了让输出简洁，`G_D` 不直接保存完整铁路语义，而是保存成 task output：

```text
G_D = targets
```

当前有两个 task：

```text
task 0 = 在 event pool 中生成 delay
task 1 = 在 section pool 中生成 speed_limit / interruption
```

一个样本的目标大致长这样：

```json
{
  "0": {
    "count": 1,
    "anchor_index": [15],
    "params": [[0.0069]]
  },
  "1": {
    "count": 2,
    "anchor_index": [3, 8],
    "params": [
      [0.35, 0.04, 0.22857143],
      [0.42, 0.02, 0.0]
    ]
  }
}
```

含义是：

- task 0 生成 1 个晚点，落在 event pool 第 15 行，参数是归一化延误时长。
- task 1 生成 2 个区间类扰动，落在 section pool 第 3 行和第 8 行，参数是开始时间、持续时间、限速值。
- 限速值解码后如果低于中断阈值，就解释为中断。

这里的 `anchor_index` 只是候选池行号。真实车次、站名、区间名不进入 VAE，只保存在 profile 和 RailDisruptLab 的解码上下文里。

### 4.3 辅助扰动图 R

`G_D` 为了生成简洁，把发生时间、持续时间等内容压进了扰动节点参数里。这样做便于输出，但会弱化扰动之间的关系表达。

现实中，扰动之间经常存在组合效应：时间重叠、空间接近、同一线路附近、一个扰动引发另一个扰动的调整压力。为了让模型更重视这些关系，训练样本里额外保存辅助扰动关系：

```text
R = target_relations
```

当前关系特征包括：

```text
[
  same_anchor,
  spatial_near,
  time_overlap,
  time_gap_norm,
  delay_section_route_near,
  same_time_neighborhood
]
```

`R` 的定位很明确：

- 训练时使用，帮助 target encoder 更完整地理解真实答案。
- 生成时不直接输出，避免生成结果变复杂。
- 最终是否有连锁效果，仍通过解码后的扰动和 MILP 求解来验证。

所以三张图的关系是：

```text
C   = 全局背景和候选空间
G_D = 要生成的扰动答案
R   = 训练时补充 G_D 内部关系
```

## 5. 设计原则：输入充分，输出精简

当前工程采用的思路可以概括为：

```text
训练时尽量让模型看见充分结构；
生成时只让模型输出最小扰动事实。
```

这不是单纯为了省字段，而是为了让生成目标更稳定。

对本项目来说，真正需要模型决定的是：

```text
扰动有几个
扰动落在哪个候选 anchor 上
扰动参数是多少
```

这些内容组成 `G_D`。只要有 `G_D + C`，RailDisruptLab 就能确定性地恢复业务扰动，并重新计算扰动之间的关系。因此，`R` 不适合作为最终生成对象。

如果让模型同时生成 `G_D` 和 `R`，会引入不必要的复杂度。比如模型可能生成了两个时间上不重叠的扰动，却又生成 `time_overlap = 1`。这种冲突本来不应该交给模型处理，因为 `time_overlap` 可以由扰动参数直接算出来。

所以当前设计里：

```text
G_D 是唯一生成目标；
R 是训练阶段的辅助关系；
生成后如果需要 R，再由 G_D + C 重新推导。
```

这样做有三个好处：

- 输出接口简洁，只包含必须生成的事实。
- 派生关系不重复生成，避免和扰动事实互相矛盾。
- 训练阶段仍能利用 `R` 强化扰动之间时空关系和组合关系的学习。

这也解释了为什么 `R` 会进入训练样本，却不会进入 `vae_math_generated_graph`。它的作用不是“多生成一张图”，而是给模型一个更清楚的训练视角。

这条原则有边界：输出不能精简到无法解码，输入也不能让 decoder 依赖生成阶段拿不到的信息。当前工程里，生成阶段只依赖 `C`，decoder 只输出 `G_D`；`R` 只作为训练辅助关系使用，因此符合这个边界。

## 6. 图是怎么生成的

图生成不是让模型自己从原始 Excel 里找规律，而是先由 RailDisruptLab 把铁路对象整理成统一图格式。

完整链路是：

```text
计划时刻表 + 里程表
  -> BaseContext

BaseContext + scenario config
  -> disturbance_graph
  -> typed_vae_learning_graph
  -> vae_math_context_graph + vae_math_learning_sample
```

### 6.1 先生成 BaseContext

`prepare` 阶段读取计划时刻表和区间里程表，生成项目共享的 `BaseContext`。

`BaseContext` 里已经整理好：

- event anchors：每个车次在每个站的到达/出发事件。
- section anchors：每个运行区间。
- station order、section order、train route 等基础关系。

这一层仍然有铁路语义，供 RailDisruptLab 使用。VAE 不直接读取它。

### 6.2 从 BaseContext 生成上下文图 C

上下文图只依赖 `BaseContext`，不依赖某个具体扰动样本。

生成 pool 时：

```text
所有 event anchor   -> event pool 的每一行
所有 section anchor -> section pool 的每一行
```

生成 edge 时：

```text
event-event:
  同车、同站，或计划时间差在窗口内
  每个 event 只保留 top_k 个更相关的候选关系

section-section:
  共享端点、相邻区间，或区间顺序差在窗口内

event-section:
  事件站点是区间端点，或该车次线路经过该区间
```

然后把节点和边的铁路属性全部转成归一化数值。比如一天内时间会除以 24 小时，车次序号、站点序号、区间序号会除以各自最大值，限速会除以默认最高速度。

这一步的结果就是共享 `context.json`。

### 6.3 从 scenario 生成目标图 G_D

每个场景 YAML 会先转成标准 `disturbance_graph`。这个图里有扰动节点，也有扰动落在哪个 anchor 上的 role edge。

再生成 `G_D` 时，会按扰动类型分流：

```text
delay:
  role edge 指向 event anchor
  event anchor id -> event pool 行号 anchor_index
  delay_seconds -> delay_seconds_norm

speed_limit / interruption:
  role edge 指向 section anchor
  section anchor id -> section pool 行号 anchor_index
  start_time -> start_time_norm
  duration -> duration_norm
  speed_limit -> speed_limit_norm
```

所以 `G_D` 不是重新发明一套业务格式，而是把已经合法的铁路扰动压缩成模型可学习的 task 监督。

### 6.4 从 disturbance_graph 生成辅助关系 R

辅助关系 `R` 也是从真实扰动图推导出来的。

对每两个真实扰动，代码会比较它们的锚点、空间距离和时间区间，生成关系特征：

```text
same_anchor
spatial_near
time_overlap
time_gap_norm
delay_section_route_near
same_time_neighborhood
```

这相当于把“扰动之间可能连锁”这件事显式告诉训练阶段。它不进入最终生成结果，只帮助模型更好地理解真实样本中的组合关系。

### 6.5 多个样本共享同一张 C

同一个 project 下，训练 scenario set 共用同一份 `BaseContext`。因此每个样本导出的上下文图必须完全一致。

工程里会检查这一点：

```text
第一个样本生成 context_graph
后续样本生成的 context_graph 必须与它相同
否则报错
```

然后所有样本只保存自己的 `G_D / R`。这能避免重复保存大量候选节点和候选边，也让 VAE 明确知道：背景不变，扰动答案在变。

## 7. 文件里实际长什么样

训练数据由一份共享上下文和多份样本组成：

```text
graph/context.json
graph/samples/*.json
graph/dataset_profile.json
```

`context.json` 是上下文图 `C`：

```json
{
  "schema_version": 1,
  "graph_type": "vae_math_context_graph",
  "decode_handle": {},
  "rules": {
    "pools": [],
    "edge_types": [],
    "tasks": [],
    "target_relation_feature_dim": 6
  },
  "graph": {
    "pool_x": {},
    "edges": []
  }
}
```

`samples/*.json` 是某个训练样本的答案 `G_D / R`：

```json
{
  "schema_version": 1,
  "graph_type": "vae_math_learning_sample",
  "context_ref": "context.json",
  "supervision": {
    "targets": {},
    "target_relations": []
  }
}
```

`dataset_profile.json` 是解释文件，不是 VAE 输入。它保存 pool 行号到 anchor id 的映射、feature 名称、task 说明、source config、decode contract 和 debug 信息。

## 8. GNN 在这里做什么

GNN 的作用不是替代 VAE，而是把图结构整理成 VAE 能用的表示。

过程可以简单理解为：

```text
节点原始特征 -> 节点向量
边原始特征   -> 边向量
节点沿边互相传消息
每个节点更新自己的表示
最后把节点和边汇总成 context 向量
```

当前实现里，event、section 和 edge 会先被投影到同一个 `hidden_dim`。然后按 edge type 分别传消息：

```text
source_h + target_h + edge_h -> message
message 按 target 聚合
old_node_h + 聚合消息 -> new_node_h
```

如果边是无向的，就同时做反向消息。event-event、section-section、event-section 三类边不共享 message layer，因为它们表达的关系不同。

经过若干轮 message passing 后：

- event 节点不只知道自己的时间、车次、站点，也吸收了附近事件的信息。
- section 节点不只知道自己的里程和运行时间，也吸收了相邻区间和相关事件的信息。
- context 向量不只是节点均值，而是经过关系聚合后的运行图摘要。

默认配置下：

```text
hidden_dim = 64
2 个 pool + 3 类 edge
context_dim = 64 * 5 = 320
```

## 9. VAE 在这里做什么

VAE 的作用是学习“在给定上下文 C 下，扰动答案 G_D 更可能是什么样”。

训练时模型看到：

```text
C + G_D + R
```

其中：

- `C` 经过 GNN 编码成 context。
- `G_D` 编码成真实扰动数量、锚点位置和参数。
- `R` 编码成真实扰动之间的辅助关系。

训练目标不是让模型背答案，而是让它学到一个能生成类似答案的潜在空间。通俗地说：

```text
posterior 看过真实答案，知道这类样本大概藏在潜在空间哪里
prior 只看上下文，学习在没有答案时应该从哪里采样
decoder 从潜在变量和上下文里还原扰动答案
```

生成时没有真实答案，所以只走：

```text
C -> prior -> z -> decoder -> task_outputs
```

这就是为什么训练样本需要 `G_D / R`，而生成阶段只需要 `C`。

## 10. Decoder 输出什么

decoder 对每个 task 做三件事：

```text
1. 预测 count
2. 为每个 slot 选择 anchor_index
3. 为每个 slot 生成 params
```

task 绑定了目标 pool：

```text
task 0 -> event pool
task 1 -> section pool
```

所以 decoder 不会在整张图里乱选。task 0 只能在 event anchor 里选晚点位置，task 1 只能在 section anchor 里选限速或中断位置。

锚点选择的做法是：

```text
decoder 生成 slot query
slot query 与目标 pool 的节点向量做打分
分数最高的候选行号就是 anchor_index
```

参数生成则是直接输出数值向量，再由规则修正边界。例如：

- 晚点参数约束在合法范围内。
- 区间扰动的开始时间和持续时间不能超过 24 小时窗口。
- 限速值低于阈值时解码成中断。

VAE 只生成数学输出，铁路语义解释仍由 RailDisruptLab 完成。

## 11. 训练时怎么判断对错

训练时 decoder 会输出一份预测答案，和真实 `targets` 比较：

```text
count   是否接近真实扰动数量
anchor  是否选中真实候选锚点
params  是否接近真实扰动参数
latent  生成用的 prior 是否逐渐贴近看过答案的 posterior
```

不需要把概率公式背下来。这里的关键是：

- posterior 因为看过答案，训练早期会更准。
- prior 只能看上下文，生成时只能靠它。
- KL 项的作用就是让 prior 慢慢学会靠近 posterior。
- reconstruction loss 的作用是让 decoder 能从潜在变量和上下文还原出扰动答案。

最后生成时，模型才能在只给上下文的情况下生成新的扰动组合。

## 12. 生成结果如何回到铁路问题

VAE 输出的是：

```json
{
  "schema_version": 1,
  "graph_type": "vae_math_generated_graph",
  "decode_handle": {},
  "task_outputs": {
    "0": {
      "count": 1,
      "anchor_index": [15],
      "params": [[0.0069]]
    },
    "1": {
      "count": 1,
      "anchor_index": [3],
      "params": [[0.35, 0.04, 0.22857143]]
    }
  }
}
```

它还不是最终场景文件。后续流程是：

```text
vae_math_generated_graph
  -> disturbance_graph
  -> scenario config
  -> build / solve / analyze
```

这一步保留在 RailDisruptLab 中，是为了保证：

- anchor_index 能被还原成真实 event/section anchor。
- 参数能被解释成延误秒数、限速开始时间、持续时间和限速值。
- 中断、时间窗、速度阈值等业务规则由同一套代码处理。
- 生成样本最终能进入 MILP 验证，而不是停留在模型输出层。

## 13. 这个设计的核心

一句话概括：

```text
用图把不规整的铁路扰动样本整理成可学习的数学对象，
再用 GNN+VAE 在这个对象上学习和生成，
最后把生成结果交还给确定性的铁路规则和 MILP 流程验证。
```

创新点不在“用了 VAE”或“用了 GNN”，而在中间这层图建模：

- 把原计划运行图做成共享上下文 `C`。
- 把扰动本体压缩成简洁的目标图 `G_D`。
- 用辅助关系 `R` 补回扰动之间的时空关联。
- 用 task 规则约束生成空间，让模型只在合法候选池中生成。
- 让模型输出保持数学化、简洁化，铁路语义只在 RailDisruptLab 中解释。

这样后续无论换 VAE、换 GNN，还是增加新的扰动类型，主边界都比较清楚。
